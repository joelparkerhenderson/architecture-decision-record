# AWS 架構決策記錄流程

https://docs.aws.amazon.com/prescriptive-guidance/latest/architectural-decision-records/adr-process.html

架構決策記錄（Architectural Decision Record, ADR）是一份文件，描述團隊針對計畫建置之軟體架構中某個重要面向所做的選擇。每份 ADR 描述架構決策、其背景脈絡及其後果。ADR 具有狀態，因此遵循一個生命週期。ADR 的範例請參閱附錄。

ADR 流程的產出是一組架構決策記錄的集合。這個集合構成決策日誌（Decision Log）。決策日誌提供專案的背景脈絡以及詳細的實作與設計資訊。專案成員可以瀏覽每份 ADR 的標題來快速了解專案脈絡，也可以深入閱讀 ADR 以了解專案實作與設計選擇的細節。

當團隊接受一份 ADR 後，該 ADR 即成為不可變更的。若新的洞見需要不同的決策，團隊會提出一份新的 ADR。當團隊接受新的 ADR 時，它將取代先前的 ADR。

## ADR 流程的範圍

專案成員應針對每一個對軟體專案或產品具有架構重要性的決策建立 ADR，包括以下面向（Richards and Ford 2020）：

* 結構（例如微服務等模式）

* 非功能性需求（安全性、高可用性與容錯性）

* 依賴關係（元件之間的耦合）

* 介面（API 與已發佈的契約）

* 建構技術（函式庫、框架、工具與流程）

* 功能性與非功能性需求是 ADR 流程最常見的輸入。


## ADR 內容

當團隊識別出需要建立 ADR 時，某位團隊成員會根據專案統一的範本開始撰寫 ADR。（範本範例請參閱 ADR GitHub 組織。）範本簡化了 ADR 的建立過程，並確保 ADR 擷取所有相關資訊。每份 ADR 至少應定義決策的背景脈絡、決策本身，以及該決策對專案及其交付物的後果。（這些區段的範例請參閱附錄。）ADR 結構最強大的面向之一在於它聚焦於決策的原因，而非團隊如何實作。理解團隊為何做出該決策，能讓其他團隊成員更容易採納決策，也能防止未參與決策過程的其他架構師在未來推翻該決策。


## ADR 採用流程

每位團隊成員都可以建立 ADR，但團隊應建立 ADR 的所有權定義。每位身為 ADR 擁有者的作者應積極維護並溝通 ADR 內容。為釐清此所有權，本指南在以下章節中將 ADR 作者稱為 ADR 擁有者。其他團隊成員隨時可以對 ADR 做出貢獻。若 ADR 內容在團隊接受之前有所變更，擁有者應核准這些變更。

在團隊識別出架構決策及其擁有者後，ADR 擁有者在流程開始時提供狀態為**已提議（Proposed）**的 ADR。處於已提議狀態的 ADR 已準備好進行審查。

ADR 擁有者隨後啟動 ADR 的審查流程。ADR 審查流程的目標是決定團隊是否接受 ADR、判定其需要修改，或拒絕該 ADR。專案團隊（包括擁有者）審查 ADR。審查會議應以專門的時段閱讀 ADR 開始。平均而言，10 到 15 分鐘應已足夠。在此期間，每位團隊成員閱讀文件，並新增評論和問題以標記不清楚的主題。閱讀階段結束後，ADR 擁有者逐一唸出並與團隊討論每則評論。

若團隊發現需要改善 ADR 的行動項目，ADR 的狀態維持在**已提議（Proposed）**。ADR 擁有者撰寫行動項目，並與團隊協作，為每個行動項目指派負責人。每位團隊成員都可以貢獻並解決行動項目。重新安排審查流程是 ADR 擁有者的責任。

團隊也可以決定拒絕 ADR。在此情況下，ADR 擁有者新增拒絕原因，以避免未來就同一主題重複討論。擁有者將 ADR 狀態變更為**已拒絕（Rejected）**。

若團隊核准 ADR，擁有者新增時間戳記、版本及利害關係人清單。擁有者隨後將狀態更新為**已接受（Accepted）**。

ADR 及其形成的決策日誌代表團隊所做的決策，並提供所有決策的歷史紀錄。團隊在可能的情況下，於程式碼審查和架構審查時以 ADR 作為參考。除了進行程式碼審查、設計任務和實作任務外，團隊成員也應在產品的策略性決策中參考 ADR。

作為良好實踐，每次軟體變更都應經過同儕審查，並至少需要一次核准。在程式碼審查過程中，審查者可能發現違反一或多份 ADR 的變更。在此情況下，審查者要求程式碼變更的作者更新程式碼，並分享 ADR 的連結。當作者更新程式碼後，經同儕審查者核准並合併至主要程式碼庫。


## ADR 審查流程

團隊在接受或拒絕 ADR 後，應將 ADR 視為不可變更的文件。對現有 ADR 的變更需要建立新的 ADR、為新 ADR 建立審查流程，並核准該 ADR。若團隊核准新的 ADR，擁有者應將舊 ADR 的狀態變更為**已取代（Superseded）**。
