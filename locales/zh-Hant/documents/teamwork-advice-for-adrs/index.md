# ADR 的團隊合作建議

如果你正在考慮在團隊中使用決策記錄（Decision Records），以下是我們在與許多團隊合作中學到的一些建議。

你有機會透過與隊友一起討論「為什麼」來引導他們，而不是強制規定「是什麼」。例如，決策記錄是團隊更聰明地思考和更好地溝通的方式；如果決策記錄只是事後被迫填寫的文書工作，那就沒有價值了。

有些團隊非常偏好使用「decisions」這個名稱，而不是縮寫「ADRs」。當某些團隊使用「decisions」作為目錄名稱時，就彷彿靈光一現，團隊開始在目錄中放入更多資訊，例如供應商決策、規劃決策、排程決策等。所有這些類型的資訊都可以使用相同的範本。我們推測，人們透過完整詞彙（「decisions」）比透過縮寫（「ADRs」）學習得更快，而且當移除「record」這個詞時，人們更有動力撰寫進行中的文件，此外一些開發人員和一些管理者不喜歡「architecture」這個詞。

理論上，不可變性（immutability）是理想的。但在實務中，可變性（mutability）對我們的團隊效果更好。我們將新資訊插入到既有的 ADR 中，加上日期戳記，並註明該資訊是在決策之後才取得的。這種方式會產生一份我們都可以更新的「活文件（living document）」。典型的更新情境包括：因為新隊友帶來的資訊、新的產品方案、我們實際使用的真實結果，或事後第三方的變更，例如供應商能力、定價方案、授權合約等。
