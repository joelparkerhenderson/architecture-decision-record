# 如何開始使用 ADR

要開始使用 ADR，請與你的團隊成員討論以下幾個面向。

決策識別（Decision identification）：

  * 這個 AD 有多緊急、多重要？

  * 它需要現在就做出決定，還是可以等到掌握更多資訊後再決定？

  * 個人和團隊的經驗，以及公認的設計方法與實踐，都有助於識別決策。

  * 理想情況下，應維護一份決策待辦清單，與產品待辦清單互相搭配。

決策制定（Decision making）：

  * 決策制定技術有許多種，包括通用的方法和軟體架構專用的方法，例如對話映射（dialogue mapping）。

  * 群體決策（group decision making）是一個活躍的研究主題。

決策實施與執行（Decision enactment and enforcement）：

  * AD 被應用於軟體設計中；因此，它們必須傳達給系統的利害關係人（stakeholders），並獲得資助、開發和營運該系統的人員的認可。

  * 架構可見的編碼風格（architecturally evident coding styles）以及聚焦於架構關注點和決策的程式碼審查（code reviews），是兩種相關的實踐方式。

  * 在軟體演進（software evolution）過程中對軟體系統進行現代化改造時，AD 也需要被重新考量。

決策共享（Decision sharing，選擇性）：

  * 許多 AD 會在不同專案中重複出現。

  * 因此，過去決策的經驗（無論好壞），在採用明確的知識管理策略時，都可以成為有價值的可重用資產。

決策文件化（Decision documentation）：

  * 有許多用於記錄決策的範本和工具。

  * 參見敏捷社群的做法，例如 M. Nygard 的 ADR。

  * 參見傳統軟體工程和架構設計流程，例如 IBM UMF 建議的表格佈局，以及 CapitalOne 的 Tyree 和 Akerman 所提出的方法。
