# 架構決策記錄（Architecture Decision Record, ADR）範本 <!-- 替換為 ADR 標題 -->

這是 EdgeX Foundry ADR 的範本。

來源：https://docs.edgexfoundry.org/2.3/design/adr/template/


### 提交者

列出 ADR 提交者。

格式：

- 姓名（組織）


## 變更日誌

列出文件的變更，包括狀態、日期和 PR 網址。

狀態為以下之一：待定（pending）、已核准（approved）、已修訂（amended）、已棄用（deprecated）。

日期為 ISO 8601（YYYY-MM-DD）格式字串。

PR 是提交變更的拉取請求（Pull Request），包含差異（diff）、貢獻者和審查者等資訊。

格式：

- \[ADR 的狀態，例如 approved、amended 等\]\(拉取請求的網址\) YYYY-MM-DD


## 參考的使用案例

列出所有相關的使用案例（Use Case）／需求文件。

ADR 至少需要一個相關的、已核准的使用案例。

格式：

- \[使用案例名稱\]\(網址\)

如果 ADR 未涵蓋某個使用案例的所有需求，請新增說明。


## 背景脈絡

描述：

- 該設計為何具有架構上的重要性——值得撰寫 ADR（而非僅以簡單的 issue 和 PR 來修復問題）

- 高層級的設計方法（詳細內容在下方的提議設計中說明）


## 提議設計

設計的細節（盡可能不涉及實作層面）。

概述：

- 將受影響（變更）的服務/模組

- 將新增的服務/模組

- 模型和 DTO 的影響（變更/新增/移除）

- API 的影響（變更/新增/移除）

- 一般組態的影響（建立新區段、變更/新增/移除）

- DevOps 的影響


## 考量事項

記錄在 ADR 討論過程中出現的替代方案、疑慮、附帶或相關議題、問題。

說明是否/如何解決或緩解這些問題。


## 決策

記錄任何已達成共識的重要實作細節、注意事項、未來考量、剩餘或延後的設計議題。

記錄提議設計未滿足的任何需求部分。


## 其他相關 ADR

列出任何相關的 ADR——例如某功能子元件的設計決策、因本設計而被棄用的設計等。

格式：

- \[ADR 標題\]\(網址\) - 相關性說明


## 參考資料

列出額外的參考資料。

格式：

- \[標題\]\(網址\)
