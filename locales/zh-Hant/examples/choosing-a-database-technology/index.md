# 架構決策記錄：選擇資料庫技術

## 狀態

已接受

## 背景

我們正在設計一個新應用程式，需要以可擴展且高效的方式儲存和擷取資料。我們已識別出三種常用的資料庫技術類型：關聯式資料庫（Relational Database）、文件資料庫（Document Database）和事件資料庫（Event Database）。

關聯式資料庫以固定綱要（Schema）的資料表儲存資料，並強制執行嚴格的資料完整性約束。它們適用於需要複雜資料關聯和交易處理的應用程式。範例包括 MySQL、PostgreSQL 和 Oracle。

文件資料庫以類 JSON 的文件格式儲存資料，且不需要綱要（Schema-less）。它們非常適合需要靈活資料模型和水平擴展的應用程式。範例包括 MongoDB、Couchbase 和 Amazon DynamoDB。

事件資料庫以一系列事件的形式儲存資料，捕獲資料的每一次變更。它們適用於需要稽核、事件溯源（Event Sourcing）和複雜資料處理的應用程式。範例包括 Apache Kafka、Apache Pulsar 和 AWS Kinesis。

## 決策

在仔細評估應用程式的需求和限制後，我們決定使用文件資料庫。

## 理由

我們選擇文件資料庫的原因如下：

1. 我們的應用程式需要一個能隨時間演進的靈活資料模型。文件資料庫允許我們以無綱要的格式儲存資料，這意味著我們可以新增欄位或更改現有文件的結構，而無需修改資料庫綱要。

2. 我們的應用程式需要水平擴展以處理大量的資料和流量。文件資料庫提供內建的分片（Sharding）和複寫（Replication）支援，使我們能夠將資料分散到多台伺服器上，並處理高讀寫吞吐量。

3. 我們的應用程式需要快速且高效的資料擷取。文件資料庫提供強大的索引和查詢功能，使我們能夠快速且高效地擷取資料。

4. 我們的應用程式不需要複雜的交易處理或資料關聯。雖然關聯式資料庫在強制執行資料完整性約束和處理複雜交易方面表現出色，但我們的應用程式沒有此類需求。文件資料庫可以為我們的使用情境提供足夠的一致性和持久性保證。

## 後果

選擇文件資料庫後，我們需要投入學習和理解所選用的特定技術。此外，我們需要確保應用程式的資料模型與文件資料庫的資料模型良好配合，以最大化效能和可擴展性。

然而，我們相信使用文件資料庫的效益大於成本，且它最符合我們應用程式的需求和限制。
