## 架構決策記錄：端對端測試的瀏覽器自動化框架（Playwright vs Selenium）

### 1. **背景**

我們正在為端對端（End-to-End, E2E）測試流程選擇瀏覽器自動化框架。此框架將是 CI/CD 流程的核心，執行模擬真實使用者互動的測試。具體而言，測試將涵蓋使用者註冊/登入、檔案上傳、儀表板互動及報表下載等場景。

作為一家**新創公司**，我們專注於**敏捷開發**，需要快速迭代和演進。我們的團隊主要使用 **TypeScript** 和 **Python**，以這些語言撰寫測試的能力至關重要。此外，平台包含**互動式圖表和儀表板**，因此自動化工具必須良好支援豐富的動態使用者介面。

此任務的兩個候選方案是 **Playwright** 和 **Selenium**，各有其優勢與取捨。我們需要根據以下列出的功能和需求來評估這些框架。

### 2. **備選方案**

- **Playwright**（由 Microsoft 開發）
- **Selenium**（由 Selenium Project 開發）

### 3. **決策驅動因素**

影響我們決策的因素如下：

1. **敏捷開發**：所選工具必須支援快速、靈活的開發週期。
2. **語言支援**：我們的團隊需要同時支援 **TypeScript** 和 **Python**。
3. **互動式 UI 測試**：可靠測試互動式圖表、儀表板和動態元素的能力至關重要。
4. **執行速度**：雖非首要考量，但 CI/CD 流程中的效能仍是一個考慮因素。
5. **可擴展性**：我們近期沒有大規模擴展的計畫，但希望確保解決方案能應對未來的成長。
6. **向後相容性**：舊有系統和較舊瀏覽器的相容性目前對我們的專案並不關鍵。
7. **行動裝置測試**：雖非當前重點，但框架應能測試行動裝置響應式功能或具備此類使用情境的擴充性。
8. **多螢幕測試**：支援多螢幕配置是次要需求，尤其當我們未來擴展到測試更複雜的使用者工作流程時。
9. **檔案上傳測試**：框架必須高效處理檔案上傳，這是我們測試需求的核心要求。

### 4. **評估標準**

- **易用性**：撰寫和維護測試的容易程度如何？
- **語言支援**：框架是否支援我們團隊最常使用的 TypeScript 和 Python？
- **互動式 UI 測試**：框架處理複雜互動式使用者介面（如圖表、檔案上傳和動態資料）的能力如何？
- **CI/CD 整合**：框架與常見 CI/CD 工具和服務的整合程度如何？
- **跨瀏覽器支援**：支援哪些瀏覽器？效能表現如何？
- **效能與速度**：測試執行速度如何，特別是在 CI/CD 流程中？
- **可擴展性**：當新增更多測試或更複雜的場景時，框架的擴展能力如何？
- **社群與生態系**：框架的社群活躍度如何？是否有豐富的整合和擴充功能？

### 5. **考量**

#### 5.1 **Playwright**

##### **優點**：
1. **更聰明的本機檔案上傳 API**：Playwright 用於操作本機檔案和執行檔案上傳的 API 更簡潔直觀。這使得檔案上傳測試更容易實作和維護。
2. **語法與程式碼產生**：Playwright 擁有更簡短、精練的語法。這減少了樣板程式碼，提高了可維護性和開發效率。此外，較簡短的語法提升了 OpenAI 程式碼產生品質，使自動產生測試腳本更為容易。
3. **互動式 UI 測試**：Playwright 在測試動態互動式 Web 應用程式方面表現出色，例如具有豐富圖表、複雜使用者互動和即時更新的應用。它能非常有效地處理 WebSocket、WebRTC、Shadow DOM 及其他現代 Web 技術。
4. **跨瀏覽器支援**：Playwright 支援 **Chromium**、**WebKit** 和 **Firefox**。在這些瀏覽器上的效能表現一致，應能覆蓋我們大部分的測試需求。
5. **CI/CD 整合**：Playwright 與現代 CI/CD 平台（GitHub Actions、Jenkins 等）無縫整合。它可以在不同瀏覽器上平行執行測試，最佳化測試執行時間，適合快速開發。
6. **快速且可靠**：Playwright 整體上比 Selenium 更快，特別是在無頭模式（Headless Mode）下，且在處理非同步 Web 元素時更具韌性。

##### **缺點**：
1. **行動裝置測試受限**：雖然 Playwright 支援瀏覽器的行動裝置模擬，但缺乏像 Selenium 與 Appium 整合那樣的原生行動裝置測試能力。
2. **較小的生態系**：Playwright 仍較新且不如 Selenium 成熟。雖然它擁有快速成長的社群和良好的文件，但可能尚未具備 Selenium 提供的龐大外掛和整合生態系。
3. **瀏覽器支援受限**：雖然 Playwright 涵蓋了主要的現代瀏覽器（Chrome、Safari、Firefox），但對舊有瀏覽器（例如 Internet Explorer）的支援不如 Selenium 完善。

#### 5.2 **Selenium**

##### **優點**：
1. **更悠久的歷史與成熟度**：Selenium 已存在很長時間，擁有良好的實績紀錄。它被眾多團隊和產業廣泛使用，帶來了龐大的外掛、整合和資源生態系。
2. **跨瀏覽器與跨平台支援**：Selenium 支援**多種瀏覽器**和版本，包括 **Internet Explorer**，也可與 **Docker**、**Selenium Grid** 及**雲端服務**等各種工具整合以進行分散式測試。
3. **行動裝置測試**：Selenium 透過與 **Appium** 的整合，在行動裝置測試方面更為強大，包括 Android 和 iOS 應用程式。這使其成為行動優先或行動密集型專案的更佳選擇。
4. **多螢幕測試**：Selenium 對涉及**多螢幕**或複雜多視窗互動的場景提供更好的支援。

##### **缺點**：
1. **複雜性**：Selenium 的 API 較為冗長且明確。雖然在某些情況下這可能是優勢，但意味著需要撰寫和維護更多程式碼，這可能降低開發敏捷性——在新創公司環境中尤為重要。
2. **效能**：Selenium 的執行速度通常比 Playwright 慢，尤其在無頭模式下。隨著測試數量增加，這可能影響 CI/CD 流程。
3. **互動式 UI 測試**：在測試現代互動式 Web UI 方面，Selenium 不如 Playwright 流暢，特別是在圖表和即時資料更新方面。它需要更多的設定和處理才能可靠地與動態內容互動。

### 6. **比較摘要**

| 功能                          | Playwright                                    | Selenium                                   |
|-------------------------------|-----------------------------------------------|-------------------------------------------|
| **易用性**                    | 較簡短的語法，對現代 UI 更直觀               | 更明確，需要更多樣板程式碼               |
| **語言支援**                  | TypeScript、Python、JavaScript                | TypeScript、Python、Java、Ruby、C#        |
| **互動式 UI 測試**            | 對動態即時 UI 表現優異                        | 可處理基本 UI，但對豐富互動較冗長複雜    |
| **檔案上傳測試**              | 更聰明的檔案上傳 API                          | 較冗長、較不直觀的 API                   |
| **CI/CD 整合**                | 與 GitHub Actions、Jenkins 輕鬆整合           | 與多種 CI 工具有良好整合                  |
| **行動裝置測試**              | 受限，僅支援模擬                              | 透過 Appium 提供完整支援                  |
| **跨瀏覽器支援**              | Chromium、WebKit、Firefox                     | 完整支援主要及舊有瀏覽器                  |
| **效能**                      | 快速，針對無頭測試最佳化                      | 較慢，尤其在無頭模式下                    |
| **多螢幕測試**                | 受限                                          | 良好支援多螢幕設定                        |
| **社群與生態系**              | 持續成長，文件完善                            | 龐大、成熟、豐富的生態系                  |

### 7. **決策**

在考量需求與取捨後，**Playwright** 是更符合我們當前需求的選擇。其更聰明的本機檔案上傳測試 API、精練的語法，以及對互動式 UI 測試的強大支援，使其非常適合我們的敏捷開發週期。它同時支援 **TypeScript** 和 **Python** 對我們的團隊至關重要，而框架的現代化測試方式將使我們能撰寫簡潔、可維護的程式碼。

雖然 **Selenium** 仍是一個優秀的工具，特別是在行動裝置測試、舊有瀏覽器支援和多螢幕設定方面，但它較不適合我們當前的需求。其冗長性、較慢的效能，以及對圖表等動態 UI 較複雜的處理方式，使其對我們的使用情境而言不夠理想。

### 8. **後果**

- **即時行動**：我們將採用 **Playwright** 作為 E2E 測試工具，專注於測試涉及註冊、登入、檔案上傳、儀表板和報表下載的使用者流程。
- **長期考量**：我們將持續關注 Playwright 生態系的演進。如果我們的需求改變，特別是在行動裝置測試或舊有瀏覽器支援方面，我們可能會重新評估 Selenium。
- **培訓與文件**：開發團隊需要熟悉 Playwright 的 API，特別是處理動態 UI 和檔案上傳的部分。
- **遷移**：現有的 Selenium 測試（如果有的話）將逐步遷移至 Playwright。
