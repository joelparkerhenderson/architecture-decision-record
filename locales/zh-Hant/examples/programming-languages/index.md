## 摘要


### 議題

我們需要為我們的軟體選擇程式語言。我們有兩個主要需求：一個適合網頁應用程式的前端程式語言，以及一個適合伺服器應用程式的後端程式語言。


### 決策

我們選擇 TypeScript 作為前端語言。

我們選擇 Rust 作為後端語言。


### 狀態

已決定。我們對新的替代方案保持開放態度。


## 細節


### 假設

前端應用程式是典型的：

  * 典型的使用者和互動方式

  * 典型的瀏覽器和系統

  * 典型的開發和部署流程

前端應用程式可能會快速演進：

  * 我們希望確保快速且容易的開發、部署、迭代等。

  * 我們重視可驗證性（provability），例如型別安全（type safety），並且願意多做一些工作來實現它。

  * 我們不需要舊版相容性。

後端應用程式的要求高於一般水準：

  * 品質目標高於一般水準，特別是在可驗證性、可靠性、安全性等方面。

  * 近即時性（near-real-time）目標高於一般水準，即我們不希望因虛擬機器垃圾回收（garbage collection）而產生停頓。

  * 函數式程式設計（functional programming）目標高於一般水準，特別是在平行化（parallelization）、多核心處理和記憶體安全方面。

我們接受較慢的編譯速度，以換取編譯時安全性和執行時速度。


### 限制條件

我們有一個強烈的限制條件：語言必須可用於主要雲端服務提供商的函式服務，例如 Amazon Lambda。


### 立場

我們考慮了以下語言：

  * C

  * C++

  * Clojure

  * Elixir

  * Erlang

  * Elm

  * Flow

  * Go

  * Haskell

  * Java

  * JavaScript

  * Kotlin

  * Python

  * Ruby

  * Rust

  * TypeScript



### 論點

各語言摘要：

  * C：因安全性低而被否決；Rust 幾乎在所有方面都做得更好。

  * C++：因語言過於混亂而被否決；Rust 幾乎在所有方面都做得更好。

  * Clojure：優秀的建模能力；最佳的 Lisp 近似語言；在 JVM 上有優秀的執行時期表現。

  * Elixir：優秀的執行時期表現，包括可部署性和並行性；優秀的開發者體驗；生態系統相對較小。

  * Erlang：優秀的執行時期表現，包括可部署性和並行性；開發者體驗較具挑戰性；生態系統相對較小。

  * Elm：看起來非常有前途；IBM 正在發布具有良好成果的重要案例研究；生態系統較小。

  * Flow：對 JavaScript 的有趣改進；然而開發者正在逐漸遠離它。

  * Go：優秀的開發者體驗；優秀的並行處理；但有一系列削弱語言能力的錯誤決策記錄。

  * Haskell：最佳函數式語言；開發者社群較小；尚未達到足夠多的已發布生產環境成功案例。

  * Java：優秀的執行時期；優秀的生態系統；開發者體驗不夠理想。

  * JavaScript：有史以來最流行的語言；最廣泛的生態系統。

  * Kotlin：修復了 Java 的許多問題；JetBrains 的優秀支持；有良好的從 Java 遷移到 Kotlin 的已發布案例。

  * Python：最流行的系統管理語言；優秀的分析工具；良好的網頁框架；但已被 Google 棄用而轉向 Go。

  * Ruby：有史以來最佳的開發者體驗；最佳的網頁框架；最友善的社群；但非常慢；打包有一定難度。

  * Rust：最佳的新語言；零抽象成本的重點；並行處理的重點；然而生態系統相對較小；且對某些類型的編譯器加速有刻意的限制，例如直接記憶體存取需要明確標記為 unsafe。

  * TypeScript：為 JavaScript 加入型別；優秀的轉譯器；越來越多開發者傾向從 JavaScript 遷移到 TypeScript；Microsoft 的強力支持。

我們認為虛擬機器（VM）有一組我們目前不需要的權衡取捨，例如提供執行時期能力的額外複雜性。

我們相信我們的核心決策由兩個橫切關注點（cross-cutting concerns）驅動：

  * 要達到最快的執行時期速度和最緊密的系統存取，我們會選擇 JavaScript 和 C。

  * 要達到接近最快的執行時期速度和接近最緊密的系統存取，我們選擇 TypeScript 和 Rust。

值得一提的是，如果我們想要 VM 語言，我們會選擇以下 VM 語言和網頁框架：

  * Clojure 和 Luminus

  * Java 和 Spring

  * Elixir 和 Phoenix


### 影響

前端開發者需要學習 TypeScript。如果開發者的主要經驗是使用 JavaScript，學習曲線可能較為平緩。

後端開發者需要學習 Rust。如果開發者的主要經驗是使用 C/C++，學習曲線可能是中等的；如果開發者的主要經驗是使用 Java、Python、Ruby 或類似的記憶體管理語言，學習曲線可能是困難的。

TypeScript 和 Rust 都是相對較新的語言。這意味著許多工具尚未有這些語言的文件。例如，DevOps 管線需要為這些語言進行設定，而到目前為止，我們正在評估的 DevOps 工具都沒有這些語言的預設範例。

TypeScript 和 Rust 的編譯時間相當慢。其中部分原因可能是語言較新。我們可能需要研究如何緩解編譯時間慢的問題，例如按需編譯（compile-on-demand）、並行編譯（compile-concurrency）等。

這些語言的 IDE 支援尚未普及，也尚未達到一流水準。例如，JetBrains 銷售的 PyCharm IDE 為 Python 提供一流支援，但並未銷售具有一流 Rust 支援的 IDE；JetBrains 需要使用 Rust 外掛程式，該外掛程式提供大約 80% 的 Rust 語言支援（相對於 Python 語言支援而言）。


## 相關內容


### 相關決策

我們將傾向於選擇與這些語言一致的生態系統。

例如，我們希望選擇一個對這些語言有良好支援能力的 IDE。

例如，對於我們的前端網頁框架，我們更傾向選擇偏向 TypeScript 的框架（如 Vue），而非偏向純 JavaScript 的框架（如 React）。


### 相關需求

我們的整個工具鏈必須支援這些語言。


### 相關產出物

我們預期可能會將一些機密資訊匯出到環境變數中。


### 相關原則

三思而後行（Measure twice, build once）。我們優先考慮安全性而非速度。

執行時期比編譯時期更有價值。我們優先考慮客戶使用體驗而非開發者使用體驗。
