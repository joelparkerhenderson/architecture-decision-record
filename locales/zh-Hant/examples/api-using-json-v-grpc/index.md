# 架構決策記錄：API 使用 JSON 與 gRPC

## 狀態

已接受

## 背景

我們正在為一個將被多個客戶端使用的新服務設計 API。我們考慮了兩種實作 API 的方案：使用基於 HTTP 的 JSON，或使用 gRPC。

基於 HTTP 的 JSON 是一種廣泛使用的 API 建構方式，許多程式語言和框架都支援它。這種方式簡單、輕量且易於理解，是許多專案的良好選擇。然而，它的效率可能不如其他選項，尤其在處理大量資料時。

另一方面，gRPC 是一種較新的技術，提供了更高效的 API 建構方式。它使用二進位序列化（Binary Serialization）來傳輸資料，比使用 JSON 更快速且更緊湊。gRPC 還支援雙向串流（Bidirectional Streaming），是即時應用程式的良好選擇。

## 決策

在權衡兩種方案的優缺點後，我們決定使用 gRPC 作為我們的 API。雖然基於 HTTP 的 JSON 是較簡單的選項，但我們認為 gRPC 將為我們的服務提供更高效且更具擴展性的解決方案。我們預期 API 將處理大量資料，而 gRPC 的二進位序列化對此使用情境將更為高效。

此外，我們認為 gRPC 對雙向串流的支援將有利於我們未來可能開發的即時應用程式。

## 後果

選擇 gRPC 意味著我們需要使用與基於 HTTP 的 JSON 不同的工具和函式庫集來建構 API。這可能需要額外的時間和精力來學習和實作這些技術。此外，想要使用我們 API 的客戶端需要使用相容 gRPC 的函式庫，而這些函式庫的支援範圍可能不如基於 HTTP 的 JSON 函式庫廣泛。

然而，我們認為使用 gRPC 的優勢大於這些潛在的缺點，並且我們有信心這個決策將帶來更高效且更具擴展性的 API。
