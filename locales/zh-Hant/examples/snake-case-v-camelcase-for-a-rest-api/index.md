# 架構決策記錄：REST API 的 snake_case 與 camelCase 之爭

## 背景脈絡

在 REST API 的命名慣例中，有兩種流行的格式：snake_case 和 camelCase。snake_case 格式是名稱中的每個單字以底線分隔，而 camelCase 則是名稱的第一個單字小寫，後續單字的首字母大寫。此決策將決定 REST API 應使用哪種命名慣例。

## 決策驅動因素

- 與專案中現有命名慣例的一致性

- 與 REST API 命名慣例的業界最佳實踐保持一致

- 實作和維護的便利性

## 決策

REST API 端點將使用 snake_case 命名慣例。此選擇基於以下因素：

1. **一致性**：專案已經在所有端點使用 snake_case 命名慣例，維持此慣例以確保整個專案的一致性是有益的。

2. **可讀性和清晰度**：snake_case 慣例更具可讀性且更容易理解。底線在單字之間提供了清晰的分隔，使名稱的含義更容易解析和理解。

3. **符合業界最佳實踐**：snake_case 慣例在業界被廣泛使用，被認為是 REST API 的最佳實踐，因此是專案的良好選擇。

4. **實作和維護的便利性**：沿用現有的命名慣例更容易實作和維護，因為如果選擇新的慣例，所有現有的程式碼和文件都需要更新。

## 後果

此決策有一些潛在的後果。

* 如果加入專案的新團隊成員不熟悉 snake_case 命名慣例，可能會導致開發中的困惑和錯誤。然而，由於 snake_case 是一種被廣泛使用的慣例，這樣的風險是很小的。
