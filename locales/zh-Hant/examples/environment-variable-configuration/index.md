## 摘要


### 議題

我們希望應用程式能在產出物（Artifact）/ 二進位檔（Binary）/ 原始碼之外進行配置，使得同一個建置版本能根據部署環境的不同而有不同的行為。

  * 為了達成此目標，我們希望使用環境變數配置。

  * 我們希望透過可以進行版本控制的檔案來管理配置。

  * 我們希望提供一些開發者體驗方面的便利性，例如知道哪些項目可以配置以及相關的預設值。


### 決策

決定使用 .env 檔案，搭配相關的預設值檔案和結構描述檔案（Schema File）。


### 狀態

已決定。對於新出現的能力持開放態度進行考量。


## 細節


### 假設

我們傾向將應用程式碼和環境配置碼分離。我們假設應用程式需要在不同環境中有不同的運作方式，例如開發環境、測試環境、展示環境、正式環境等。

我們遵循業界的「12 要素應用程式（12 Factor App）」實務，甚至更進一步的「15 要素應用程式（15 Factor App）」相關實務。

我們過去許多專案都使用了 `.env` 檔案或類似的 `.env` 目錄的慣例。典型的做法是將這些檔案排除在版本控制之外，改用其他方式來部署、版本管理和維護它們。


### 限制條件

我們希望將機密資訊排除在原始碼管理（SCM）版本控制系統（VCS）之外。

我們希望與主流的軟體框架和函式庫相容。例如，Node 有一個 "dotenv" 模組可用於讀取環境變數配置。


### 立場

我們考慮了幾種方式：

  * 將配置儲存在應用程式中，例如 `config.js` 檔案。

  * 將配置儲存在環境中，例如 `.env` 檔案。

  * 從已知位置擷取配置，例如授權伺服器（License Server）。


### 論點

我們選擇了 .env 檔案方式，原因如下：

  * 這種方式廣受歡迎，包括專家在內。

  * 它遵循了 `.env` 檔案的模式，我們的團隊在許多專案中已多次成功使用。

  * 它很簡單。值得注意的是，我們目前能接受我們所看到的重大取捨，例如與授權伺服器方式相比缺乏稽核能力。


### 影響

我們需要找出一種方法，將公開的環境變數配置與機密管理（Secrets Management）分離。


## 相關資訊


### 相關決策

我們期望所有應用程式都採用此方式。

我們將計畫升級任何使用較低階方式的應用程式，例如在二進位檔或原始碼中寫死配置。

我們將維持任何使用更高階方式的應用程式現狀，例如使用授權伺服器。


### 相關需求

我們將為這些檔案新增 DevOps 能力，包括 Hook、測試和持續整合（Continuous Integration）。

我們需要對所有開發團隊成員進行此決策的教育訓練。



### 相關產出物

每個部署區域都需要有自己的 .env 檔案和相關檔案。


### 相關原則

容易復原（Easily Reversible）。


## 備註


範例檔案 `.env`：

```env
NAME=Alice Anderson
EMAIL=alice@example.com
```

範例檔案 `.env.defaults`：

```env
NAME=Joe Doe
EMAIL=joe@example.com
```

範例檔案 `.env.schema`，僅包含鍵值名稱：

```env
NAME
EMAIL
```
