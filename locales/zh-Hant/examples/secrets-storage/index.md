## 摘要


### 議題

我們需要儲存機密資訊（Secrets），例如密碼、私鑰、認證權杖等。

部分機密資訊是面向使用者的。例如，我們的開發人員希望能夠使用手機查詢某項服務的密碼。

部分機密資訊是面向系統的。例如，我們的持續交付（Continuous Delivery）流水線需要能夠查詢雲端託管服務的憑證。


### 決策

使用 Bitwarden 儲存面向使用者的機密資訊。

使用 HashiCorp 的 Vault 儲存面向系統的機密資訊。


### 狀態

已決定。我們對未來出現的新替代方案持開放態度。


## 細節


### 假設

就此目的及我們目前的狀態而言，我們重視面向使用者的便利性，例如好用的行動應用程式。

  * 我們希望確保在外出時也能快速輕鬆地存取，例如開發人員在值班進行系統可靠性工程（SRE）時的需求。

  * 我們希望能夠在選定的人員之間共享某些機密資訊，例如一個團隊。

我們並不試圖解決單一供應商的問題，例如將所有機密資訊專門儲存在 Amazon、Azure 或 Google 上。

我們不希望採用臨時性的方式，例如「記住它」、「寫在便條紙上」或「自己想辦法儲存」。

就此目的而言，我們的安全模型可以接受使用廣受認可的 COTS（Commercial Off-The-Shelf，商用現成品）供應商，例如 SaaS 密碼管理工具。


### 限制條件

目前我們希望簡單易用的方案，即不需要撰寫程式碼、不需要安裝伺服器、不需要做出重大承諾、不需要統一所有人的做法。


### 方案選項

我們考慮了：

1. 面向使用者的現成密碼管理工具：LastPass、1Password、Bitwarden、Dashlane、KeePass、pass、GPG 等。

2. 面向系統的 COTS 密碼管理工具：AWS KMS、HashiCorp 的 Vault、EnvKey、Thycotic 的 Secret Server、Devolutions Password Server、Lyft 的 Confidant。

3. 以共享為導向的方式：使用共享的 Google 文件、共享的 Slack 頻道、共享的網路資料夾等。

4. 低技術的臨時方式，例如記憶、寫便條、或依賴每位使用者自行想辦法。


### 論證

Bitwarden、LastPass、1Password 和 Dashlane 都是商用現成產品。

  * 為使用者、團隊、組織等提供類似的功能特性。

  * 具備 Windows 和 Mac 的桌面端功能，以及 Android 和 iOS 的行動端功能。

  * 提供 Chrome 和 Firefox 的瀏覽器擴充套件，用於自動表單填入等功能。

Bitwarden 相較於其他方案有兩個優勢：

  * Bitwarden 是開源的，這意味著其安全性可以接受同儕審查，而且該公司廣受注重安全性的開發者讚賞。

  * 軟體從業人員的經驗分享顯示，相較於其他方案，他們明顯偏好 Bitwarden。

一篇典型的優質介紹文章：https://jcs.org/2017/11/17/bitwarden

一個典型的對比投票網站：https://stackshare.io/stackups/bitwarden-vs-dashlane

我們暫緩 KeePass、pass、GPG 等方案，因為它們有額外的複雜性。這些對技術使用者來說都是不錯的解決方案。GPG 對於需要跨系統命令列操作能力的技術使用者尤其合適。

我們暫緩 KMS，因為它有單一供應商鎖定的問題。

我們選擇 Vault 來滿足面向系統的需求，因為其評價極為正面，而且 HashiCorp 在頂級品質的軟體和支援方面有著優秀的歷史紀錄。

我們否決了共享方式，例如透過共享文件、共享頻道、共享網路資料夾等。這些方式無法提供我們想要的安全性品質。

我們否決了臨時性的低技術方式，因為我們都同意這不是一條長期可行的路。


### 影響

開發人員可能需要在兩個地方追蹤機密資訊：Bitwarden 用於面向使用者的存取，Vault 用於面向系統的存取。


## 相關資訊


### 相關決策

選擇 CI/CD 伺服器的決策必須包含存取機密資訊能力的驗證。

我們需要決定如何管理這些機密資訊，包括政策、輪換、組織架構等方面。


### 相關需求

機密資訊將有合規性、稽核，以及人力資源入職/離職的相關需求。


### 相關產出物

我們預期可能會將部分機密資訊匯出為環境變數。


### 相關原則

容易可逆。

容易並行，即同時使用多種密碼管理工具很容易。

試用成本低，即有免費試用期且無需承諾。


## 備註

以下為評估筆記。這些備註均來自各 DevOps 討論區的公開評論。


### HashiCorp 的 Vault

Vault 正是你在這裡需要的東西。

不過不要直接將 Vault 投入生產環境，先在測試環境中架設，因為 HashiCorp 的文件可能相當欠缺，即使他們的產品很出色。

學習曲線非常陡峭，且架設並非輕而易舉。

初始設定有些麻煩。但這非常值得，而且社群會提供足夠的支援讓你順利上手。

文件很糟糕，但網路上有很多人分享設定經驗的指南，如果你把其中幾篇整合起來，就能得到一個可運作的設定。

初始設定需要調整他們的 helm charts（vault 和 consul）。雖然技術上你可以使用很多其他後端，但我真的非常不建議這樣做。如果你沒有大量資料要儲存，後端/consul 可以非常小型。

一定要熟悉使用 CLI，因為 GUI 更像是他們企業版的概念驗證/廣告入口。

無法直接「填滿」這件事是個痛點。例如，如果你有 5 個欄位，你需要手動為每個項目逐一新增每個欄位。所以這不像是你為特定類別預先定義欄位，然後為該類別中的所有項目填入那些欄位，而更像是「你每次都要從頭產生所有東西」，這（在我看來）很令人困擾。

你可能也想看看 goldfish 作為 vault 之上的 UI。它讓你的團隊更容易接受使用 vault。他們也有演示環境。1. 架設 consul。2. 架設指向 consul 的 vault。3. 架設指向 vault 的 goldfish。4. 設定某個 cron 任務來執行 consul snapshot 進行備份。



### LastPass

LastPass Teams。我們在使用，有自訂範本、ACL，在我看來沒有缺少什麼。

我在我的組織中導入了 LastPass，給它一個 C+/B- 的評價。最近最大的問題是可靠性不足。在過去 90 天裡，有多個小時的時間，保管庫被迫進入離線模式。這對我的組織來說並不理想，因為我們在 20 多個共享資料夾中儲存了超過 4,000 個密碼。可以想像，在這麼多密碼的情況下，每天至少會有幾個被更新或新增。我們有一個災難恢復計畫，以應對問題持續超過一兩個小時的情況：一個腳本每晚會對保管庫進行簽名和加密的 CSV 匯出，可以匯入到 KeePass 中。

LastPass 曾有未報告的服務降級事件：登入「有效」但不會拉取網站資料、管理面板中的隨機功能故障，以及未正確共享新頂層共享資料夾的金鑰。我有一個專門的「金鑰推送」/備份使用者，加入了每個群組。通常以該使用者登入可以修復任何金鑰共享問題，但在服務降級時就不行了，即使狀態頁面顯示正常......

就整合而言，如果你有適當的 ACL 並採用最小權限模型（例如，如果使用者對某個條目或資料夾同時有讀寫和唯讀權限，則只獲得唯讀權限），那整合會很容易。不幸的是，我的組織的 ACL 不是最理想的，所以我最終使用了 JSON 供應 API 和約 500 行的 Python 程式碼，因為我們數百個 ACL 的依賴性質無法很好地對應到最小權限模型。我最終取得使用者所在的所有 ACL，並進行類似依賴性遍歷的處理。

如果你的 ACL 或群組結構已經以最小權限結構為基礎建立，那麼 Windows 版的 AD/LDAP 同步工具會運作良好。

聯繫他們的銷售團隊，他們可以為你安排更長的企業版試用。確保在決定之前充分了解其限制。我們經歷了不少成長痛苦，但除了伺服器端的中斷或損害外，一切都非常順暢。


### Bitwarden

Bitwarden 有很好的工具支援（WebUI、CLI、行動端、桌面端）。可自行架設且相當容易設定。文件相當完善，且是 PrivacyTools 推薦的工具。


### EnvKey

https://www.envkey.com/ 是一個 SaaS 服務。非常容易實作、整合和管理。

功能特色：

  * 保護 API 金鑰和憑證。

  * 讓設定在各處保持同步。

  * 智慧型的端對端加密設定和機密資訊管理。

  * 防止不安全的共享和設定蔓延。

  * 幾分鐘內即可整合。

能力：

  * 在一個地方管理所有應用程式、環境和團隊的設定及存取層級。

  * 只需一個環境變數即可設定任何開發或伺服器環境。

優點：

  * 良好的首頁。

  * 清晰的價值主張。

  * 視覺上出色的網頁應用程式。

  * 優質的範例資料，例如 Algolia、AWS、Datadog、GitHub、Stripe 等。

  * 與創辦人進行了 30 分鐘的交流，討論了公司、UI 等。Dane 聽起來很有見識，對優缺點很坦誠，是一個可行的合作夥伴。

  * 該公司基本上是一家典型的 Y Combinator 公司，有 1 位創辦人。2018 年 1 月募集了 12 萬美元。

  * 重點是開發企業功能，特別是從 EnvKey 雲端託管遷移到本地部署或 BYOC（Bring Your Own Cloud，自帶雲端）。

  * 可能的發展路徑是先使用 EnvKey 以便於使用，之後（或同步）再加入 Vault。


### Lyft 的 Confidant

https://lyft.github.io/confidant/

Confidant 是一個開源的機密資訊管理服務，由 Lyft 的開發人員開發，以安全的方式提供使用者友善的機密資訊儲存和存取。

KMS 認證：Confidant 透過使用 AWS KMS 和 IAM 來解決認證的先有雞還是先有蛋的問題，允許 IAM 角色產生可由 Confidant 驗證的安全認證權杖。Confidant 還為你的 IAM 角色管理 KMS 授權，這使得 IAM 角色可以產生用於服務對服務認證的權杖，或在服務之間傳遞加密訊息。

版本化機密資訊的靜態加密：Confidant 以僅追加的方式在 DynamoDB 中儲存機密資訊，使用 Fernet 對稱認證加密技術，為每個機密資訊的每個版本產生唯一的 KMS 資料金鑰。

使用者友善的機密資訊管理網頁介面：Confidant 提供一個 AngularJS 網頁介面，讓終端使用者可以輕鬆管理機密資訊、機密資訊與服務的對應關係，以及變更歷史記錄。


### Devolutions Password Server

https://server.devolutions.net/

安全地管理和監控特權帳號及工作階段的存取。

一個全面、高度安全的密碼保管庫，讓你控制特權帳號的存取，同時為系統管理員提升整體網路可見性，並為終端使用者提供無縫體驗。

功能：集中式組織密碼保管庫、使用者專屬私人保管庫、密碼管理器、憑證注入、Active Directory 整合、基於角色的存取控制、雙因素認證、企業就緒、IP 限制、管理功能、自動密碼產生器、行動應用程式存取、密碼歷史記錄、存取報告、電子郵件通知。

  * 支援資料加密

  * 支援多種認證方案，包括 LDAP、O365 和本地使用者，並支援來自多個來源的 MFA

  * 多個儲存庫/保管庫，具有針對多個團隊的細粒度存取控制

  * 現代化的 Web UI

  * 私人憑證和連線保管庫，用於個人憑證/連線

  * iOS/Android 行動應用程式

  * 每個條目的稽核日誌，記錄誰/什麼/何時，並可選擇提示存取原因

  * 可自訂範本（雖然原生支援數百種連線類型）

  * 更多功能以及一個 Windows/Mac 桌面用戶端（Remote Desktop Manager），可與之同步，大幅擴展選項......一鍵連線

  * 定價不算太高——密碼伺服器最多 15 位使用者每年 500 美元


### Thycotic 的 Secret Server

https://thycotic.com/products/secret-server/

本地部署版功能：

  * 完全掌控你的端對端安全系統和基礎設施

  * 在你的本地資料中心或自己的虛擬私有雲實例中部署軟體

  * 滿足要求所有資料和系統必須駐留在本地的法律和法規義務

雲端版功能：

  * 軟體即服務（SaaS）模式讓你註冊後即可立即開始使用

  * 隨著成長進行彈性擴展

  * 由 Azure 提供控制和冗餘，具有 99.9% 的正常運行時間 SLA

使用者回饋：

  * 我們以前使用過該產品。它很容易被繞過，而且規則只對聰明的人有效。懶惰或不熟悉的使用者在團隊區域中很容易搞砸它。價格在你與他們交談時是可以協商的。

  * 你可以使用 SQL Express 和一台 Windows 7 主機來執行它。

  * 便宜。
