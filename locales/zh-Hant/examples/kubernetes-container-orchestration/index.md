# 架構決策記錄：Kubernetes 容器編排（Container Orchestration）

## 問題陳述

我們需要為不斷成長的雲端原生（Cloud-Native）應用程式組合選擇一個容器編排平台。我們目前的傳統平台部署速度太慢，且不夠敏捷，無法跟上我們日益增長的需求。我們正在尋找一個系統，能讓我們以最有效率的方式擴展服務，同時不犧牲敏捷性或易用性。

## 考量的替代方案

1. Docker Swarm

2. Kubernetes

3. Apache Mesos

## 所做的決策

在對每個容器編排平台進行全面分析後，我們決定採用 Kubernetes 作為最符合企業需求的選項。我們選擇 Kubernetes 的理由如下：

1. **可擴展性：** Kubernetes 獨特的設計非常適合擴展應用程式，隨著我們對可擴展性的需求隨時間演變，Kubernetes 具備內建能力來因應這些變化，不會產生任何問題。

2. **去中心化架構：** Kubernetes 的主節點-工作節點（Master-Worker）拓撲確保了去中心化架構，保證不會有單點故障（Single Point of Failure）。

3. **社群支援：** Kubernetes 擁有最大且最活躍的開源社群，這意味著它有大量的貢獻者、開發者和供應商，讓我們更容易獲得幫助和尋找資源。

4. **生態系統支援：** Kubernetes 擁有不斷成長的生態系統，包含各種第三方工具，以及與容器登錄庫（Container Registry）、CI/CD 管線（Pipeline）、資料儲存等的整合。

因此，我們決定採用 Kubernetes 作為目前和近期未來的容器編排平台。
