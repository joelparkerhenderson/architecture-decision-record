## 摘要


### 議題

我們希望使用 CSS 框架來建立我們的 Web 應用程式：

  * 我們希望使用者體驗在所有主流瀏覽器和螢幕尺寸上都能快速且可靠。

  * 我們希望能在設計、版面配置、UI/UX 等方面進行快速迭代。

  * 我們希望建立響應式應用程式，特別是針對行動裝置的較小螢幕、4K 寬螢幕等較大螢幕，以及可旋轉顯示器等動態螢幕。


### 決策

決定採用 Bulma。


### 狀態

決定採用 Bulma。對新出現的 CSS 框架選擇保持開放態度。


## 詳細內容


### 假設

我們希望建立現代、快速、可靠、響應式等特性的 Web 應用程式。

典型的現代 Web 應用程式正在減少/淘汰 jQuery 的使用，原因有多個：

  * 現代 JavaScript 正逐步引入許多 jQuery 曾提供的功能，因此 jQuery 的需求降低，且有更好/更快/更小的模組提供特定實作

  * jQuery 的廣泛做法是直接操作 DOM，這對現代 JavaScript 框架（如 React、Vue、Svelte）而言是一種反模式（Anti-pattern）

  * jQuery 如果被載入兩次等情況會互相衝突


### 限制

如果我們選擇使用 jQuery 的 CSS 框架，那麼我們就必須引入 jQuery。例如，Semantic UI 使用 jQuery，而 Tachyons 不使用。

如果我們選擇極簡的 CSS 框架，那麼我們就放棄了可能現在或很快需要的框架元件。例如，Semantic UI 提供圖片輪播，而 Tachyons 不提供。


### 候選方案

我們考慮過不使用框架。這仍然是可行的，特別是因為 CSS Grid 提供了我們專案所需的大部分功能。

我們透過快速初篩考慮了許多 CSS 框架：Bootstrap、Bulma、Foundation、Materialize、Semantic UI、Tachyons 等。我們選出兩個進行深入評估的是 Semantic UI（因為它有最語意化的方式）和 Bulma（因為它有最輕量的方式且提供我們目前需要的元件）。

我們考慮了 Semantic UI。它提供許多元件，包括我們專案需要的：標籤頁、網格、按鈕等。我們以兩種方式對 Semantic UI 進行了試驗：使用典型的 CDN 檔案，以及使用 NPM 套件。我們在靜態 HTML 頁面中成功使用了 Semantic UI，但在限定時間內未能成功地在 JavaScript SPA（單頁應用程式）中建構（主要因為 jQuery 載入問題）。我們發現其他開發者一直在要求 Semantic UI 開發者建立無 jQuery 版本，原因與我們相同。其他開發者多年來一直在請求無 jQuery 版本，但開發者拒絕了，並表示任何無 jQuery 版本都太難撰寫，例如～「Semantic UI 專案有超過 22,000 個使用 jQuery 的接觸點」。

Semantic 的範例：

```html
<div class="ui top attached tabular menu">
  <a class="item">Alpha</a>
  <a class="item">Bravo</a>
</div>
```

我們考慮了 Bulma。Bulma 擁有許多與 Semantic UI 類似的功能，但沒有那麼多精細的元件。Bulma 使用現代技術建構，例如不使用 jQuery。Bulma 有一些第三方元件，其中一些我們可能會使用。


Bulma 的範例：
```html
<div class="tabs">
  <ul>
    <li><a>Alpha</a></li>
    <li><a>Bravo</a></li>
  </ul>
</div>
```


### 論證

如上所述。

具體來說，Semantic UI 在技術方面（即如此多的 jQuery 接觸點）和領導方面（即對無 jQuery 版本直接拒絕，而不是嘗試制定路線圖、持續改進或募資等）似乎都有警示訊號。


### 影響

如果我們找到一個好的無 jQuery CSS 框架，這整體上是有益且正面的。
